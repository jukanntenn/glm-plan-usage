    ## Cache behavior requirements (D-06, D-07):
    - Cache enabled + fresh entry exists → returns cached value (cache hit)
    - Cache enabled + no entry exists → fetches from API and caches result
    - Cache enabled + entry exists but TTL expired → ignores cache, fetches fresh
    - Cache disabled → always fetches from API, does not update cache
    - API fetch fails but stale cache exists → returns stale cache as fallback
    - API fetch fails and no cache exists → returns None
  </behavior>
  <action>
1. In the same `#[cfg(test)] mod tests` block:
   - Add tests for each cache behavior case listed above
   - To test fallback behavior: You can make `GlmUsageSegment::get_usage_stats()` testable by refactoring slightly to accept a closure or function pointer that fetches stats, or use a test double approach. No new dependencies needed (D-36, from user discretion).
   - The cache uses `Arc<Mutex<Option<CacheEntry>>>` — in tests you can directly manipulate the cache state to set up preconditions.
   - For expired cache tests: Create an entry with a timestamp that is older than TTL by setting `Instant::now() - Duration::from_secs(ttl + 1) — this is straightforward without flaky timing-dependent sleeps.
   - Keep all tests self-contained and independent (no shared state between tests).
2. Ensure all five required behaviors from D-07 are covered by separate tests (or one table-driven test).
</action>
