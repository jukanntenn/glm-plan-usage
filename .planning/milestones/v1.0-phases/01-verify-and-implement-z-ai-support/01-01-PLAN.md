<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env vars → API client | Untrusted `ANTHROPIC_BASE_URL` and token enter routing/auth logic here |
| API response → serde models | Remote monitor response can differ or be malformed |
| API client → verification artifact | Captured evidence must be redacted and must not overstate compatibility |
| API client → status segment | Fetch/parsing errors cross into user-visible status rendering |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | I | tests/docs/artifacts | mitigate | Keep tokens in env vars only; use synthetic base URLs in tests and redact all secrets in the verification artifact. |
| T-01-02 | S/T | `src/api/client.rs` | mitigate | Fail closed on unsupported hosts and add table-driven resolution tests for both CN and z.ai exact paths. |
| T-01-03 | T/D | `src/api/types.rs`, `src/api/client.rs` | mitigate | Parse with typed structs, add normalization only if evidence requires it, and preserve graceful fallback if parsing path is touched. |
| T-01-04 | D | `src/core/segments/glm_usage.rs` | mitigate | Keep cache fallback or `None` behavior so routing/parsing failures do not crash the status line. |
| T-01-05 | I/T | `.planning/phases/01-verify-and-implement-z-ai-support/01-z-ai-verification.md` | mitigate | Record only redacted evidence, include exact conclusion language, and forbid documentation from claiming more than the artifact proves. |
</threat_model>
